# 功能需求提案：Web端一键 APK 侧载与可视化深度应用管理中心 (Web Sideloading Hub & App Manager)

## 1. 提案背景与 Star 增长策略论证

当前项目 **ATV-Super-Controller** 已经具备了出色的设备控制、定时任务机制，以及与 Home Assistant 集成的强大底层能力。然而，若要在 GitHub 上获得指数级的关注（显著提高 Stars 数量），我们需要直击 Android TV (ATV) / 智能电视盒子用户群体中最普遍、最痛的刚性需求，并提供极具视觉冲击力和易用性的 "Wow Factor"。

### **为什么这个功能能够带来爆发式的 GitHub Stars？**

1. **直击超级痛点 (Pain Point)**：对于广大的 ATV 玩家和普通家庭用户来说，给电视安装第三方/非商店应用一直是个巨大的折磨。传统方法繁琐且劝退大量非极客用户。提供一个 **“在 PC/手机浏览器直接把 APK 拖拽进去，电视端自动完成安装”** 的功能，将彻底消除这一门槛。
2. **极强的社区自发传播属性 (Virality)**：这属于 "Must-have"（必备）级别的提效工具。用户会因为“这款工具让给电视装软件变得跟呼吸一样简单”而主动分享并点赞 (Star)。
3. **降维打击的体验与生态融合**：市面上的 ADB 工具多为孤立的桌面客户端，而我们提供的是基于 Web 的现代化方案，并且能**与项目现有的定时任务、HA 联动等自动化能力深度融合**，打造从“配置”到“管理”再到“自动化”的完整闭环。

## 2. 核心突破：与现有特性的深度结合 (Synergy)

为了构建一个更加高效的整体，新的 App Manager 绝不能仅仅是一个“用来安装和卸载的工具”，它必须与系统现有的**核心引擎 (Core Engines)** 产生乘法效应：

### 2.1 与 Scheduled Task (定时任务引擎) 的深度融合

现有的定时任务让电视可以“按时自动做某事”。结合应用管理能力，我们可以扩展出极具价值的自动化场景：

- **“午夜电视洗澡”模式 (Auto-Cleanup & Optimization)**：新增 `force-stop` 和 `clear-cache` 动作。用户可以配置定时任务：“每天凌晨 3:00，遍历并强制停止所有第三方非白名单应用，并清理缓存”。这能彻底解决安卓电视越用越卡的顽疾！
- **定时分发/更新应用**：对于拥有多台 ATV 或需要统一部署的出租屋场景，通过定时任务在闲时夜间自动侧载/更新指定文件夹内的 APK。

### 2.2 与 Home Assistant (MQTT) 生态的无缝联动

充分利用项目现有的 HA MQTT Auto-Discovery 基础设施：

- **动态应用开关 (Dynamic App Buttons)**：App Manager 扫描到的每一个第三方应用，自动通过 MQTT 向 HA 注册为一个实体 Button/Switch。用户可以直接在 HA 仪表板里点击按钮调起电视上的某个 App。
- **存储与性能状态上报 (Sensors)**：应用管理模块定期分析 ATV 的存储空间大小、已安装 App 数量，并作为 Sensor 暴露给 HA。用户可以在 HA 中配置“电视空间不足 10% 时向手机发送通知”的自动化。

### 2.3 与动作编排 (Action Macros) 的融合

- **安装后即时联动**：利用现有的 `launch-app` 能力，支持“拖拽安装 APK -> 安装成功后自动在电视上将其唤起”。
- **高级ADB提权流 (Automated Permissions)**：部分高级工具（如投屏、按键映射工具）侧载后需要繁琐的 ADB 授权。App Manager 可以内置一键/自动化的宏指令（如自动执行 `pm grant <package> android.permission.SYSTEM_ALERT_WINDOW`），免去用户查阅文档手动输入的麻烦。
- **UI 桌面集成 (Launchpad)**：在现有的 Web 仪表板 Dashboard 中增加“收藏应用”模块。App Manager 中高频使用的应用可以一键 Pin (固定) 到主页，充当虚拟遥控器的核心启动板。

## 3. 功能模块概述

新增的 **"App Manager"** 模块将在前端和后端全面接管应用生命周期。

**核心特性：**

- **零配置一键拖拽侧载**：浏览器中拖入 `.apk` 文件即自动上传、推送到电视端并隐式静默安装，全程流畅的实时进度条展示。
- **全局应用透视与净化**：列出电视上所有已安装应用（区分系统与第三方），支持直观展示体积、版本号。
- **单应用级精细控制**：支持对任意应用执行“启动”、“强制停止”、“清除数据/缓存”和“一键卸载”，操作结果立即回传 Web UI，无需查看电视屏幕。
- **一键提取 (APK Backup)**：将电视中已安装的罕见应用反向提取 (`adb shell pm path` + `adb pull`) 下载到本地电脑。

## 4. 详细需求说明 (User Stories)

### Epic A: 飞轮核心 - 无感应用侧载与分发

- **Story A.1**：作为用户，我看到 Web UI 上有一个显眼的文件上传触发区。放入 `.apk` 文件后，系统自动校验并将其静默安装到 ATV 上。
- **Story A.2**：作为用户，我能清晰地看到状态流转提示（`解析中... -> 正在传输 50%... -> 电视端安装中... -> 成功！启动它？`），并在失败时获得人类可读的错误解释及修复建议。

### Epic B: 深度管理与仪表板融合

- **Story B.1**：作为管理强迫症，我能在一个列表里看到所有应用的“占用空间大小”和“包名”，并能多选批量“强制停止”或“卸载”无用的垃圾应用。
- **Story B.2**：作为老派用户，我遇到电视卡顿时，可以点击 Web 界面上的“一键深度清理”按钮，这会调用底层接口帮我关掉所有不在白名单的后台进程。
- **Story B.3**：作为用户，我可以在应用列表中把喜欢的应用点亮“星标”，它们会立刻出现在首页 Dashboard 的快捷遥控区，方便我随时点击启动。

### Epic C: 自动驾驶 - 调度与 HA 接入

- **Story C.1**：作为极客玩家，我在创建 Scheduled Task 计划任务时，可以在下拉菜单中选择新动作 `[Action: 清除App缓存]`，让电视保持极致流畅。
- **Story C.2**：作为全屋智能玩家，一旦通过 Web UI 装好了新的直播软件，我的 Home Assistant 界面会自动弹出一个新的控制按钮，允许我在手机 HA App 上直接遥控打开这个直播软件。

## 5. 技术架构思路与可行性评估

基于目前项目的架构体系，实现本提案是完全可行且成本低廉的，主要技术路径依赖于对现有 `adb shell` 环境和原有核心组件的编排使用。

### 后端 API 扩展 (Node.js 层)

1. **统一上传与 ADB 桥接通道**：引入 `formidable` 或 `multer` 支持跨平台大文件暂存，利用现有的底层 ADB 连接执行 `adb push` 和 `adb install -r`。
2. **应用清单引擎与解析器**：通过 `adb shell pm list packages -f -3` 以及 `dumpsys package` 提取应用元信息。
3. **控制器扩展**：向现有的 `actions/` 策略包（当前有 wake, shutdown, launch-app 等）注册新的内部 Action Handler，如 `install-app`, `uninstall-app`, `clear-cache`, `force-stop`，从而自然地适配现有的 Scheduled Tasks 引擎结构。

### 前端 UI 升级 (Web 层)

1. **集成式拖放控制台**：复用或引入轻量级的 Dropzone 组件，结合 Fetch API 原生上传机制和 CSS3 动画平滑过渡。
2. **Launchpad (火箭发射板)**：在现有的 Dashboard 视图增加一个 `GridView`，请求 `/api/apps?pinned=true` 并将图标（可通过第三方 API 或本地内置映射获取）平铺展示，点击即调用现存的 `launch-app` 路由。
